# Sentinel AI Content Detection Backend

Sentinel AI is a high-performance content detection backend designed to analyze text, images, audio, and video to determine if they were generated by artificial intelligence. The system consists of an ASP.NET Core Web API acting as the orchestrator and gateway, and a Python FastAPI machine learning microservice that runs specialized NLP classification models.

## Architectural Overview

The backend system is structured around **Clean Architecture** (Onion Architecture) to ensure separation of concerns, ease of testing, and database/framework independence.

### Project Structure & Layers
* **Domain (`/Domain`)**: Contains the core entities (such as `User`, `Content`, `AIDetectionResult`, `OtpRecord`), value objects, and repository interfaces. It has no dependencies on external libraries or other project layers.
* **Application (`/Application`)**: Defines the business use cases, repository interfaces, Data Transfer Objects (DTOs), and service contracts (e.g., `IAiDetectionService`, `IEmailService`).
* **Infrastructure (`/Infrastructure`)**: Contains database context (`ApplicationDbContext`), Entity Framework Core repository implementations, JWT/refresh token generation logic, and external integrations (such as `MailKit` SMTP email delivery).
* **Presentation (`/API`)**: The ASP.NET Core Web API controllers. It handles user requests, validation, request routing, and dependency injection configuration.
* **ML Inference (`/ML_Models`)**: A standalone FastAPI Python service hosting the PyTorch machine learning models.

## Core System Features

### 1. Multilingual Text Detection
* The backend automatically identifies the language of incoming text requests using Unicode character matching.
* Requests containing Arabic script are routed to the specialized Arabic model endpoint, while English text is routed to the English model endpoint.

### 2. Identity and Authentication Engine
* **JWT Access & Refresh Tokens**: Secured authentication using JSON Web Tokens (JWT) with long-term refresh token expiration (7 days) stored securely in the database.
* **OTP Email Verification**: Mandates email validation via One-Time Passwords (OTP) delivered via SMTP.
* **SSO Integration**: Pre-configured support for Google OAuth/SSO login and guest/anonymous temporary sessions.
* **OTP Rate Limiting**: The system prevents email spam by restricting verification requests from the same email address to once every 60 seconds.

### 3. Machine Learning Optimizations
* **Lazy Loading**: PyTorch models are loaded into system memory only when the first classification request for that language is received, decreasing system startup memory consumption.
* **Time-to-Live (TTL) Unloading**: An internal monitor unloads models from RAM if they remain idle for a configurable period (default: 10 minutes), freeing up server resources.

### 4. Background Maintenance Services
* **Expired OTP Cleanup**: An ASP.NET Core Hosted Service (`OtpCleanupService`) runs continuously in the background to purge expired OTP records from the SQL Server database periodically, preventing table bloat.

---

## Database Schema

The relational database (Microsoft SQL Server) contains the following core tables:
* **`Users`**: Stores account information, password hashes (BCrypt), Google provider details, and refresh tokens.
* **`OtpRecords`**: Temporary table storing 6-digit verification codes, creation timestamps, expiration times, and verification states.
* **`Contents`**: Stores the raw inputs (text or media paths) uploaded by users for scanning.
* **`AIDetectionResults`**: Logs the detailed detection metadata returned by the ML models, including the probability score and binary AI generation flags.

## Installation & Running Locally

### Prerequisites
* .NET 8.0 SDK
* Python 3.10 or higher
* SQL Server 2022 (or LocalDB)

### 1. Set Up and Run the ML Service
1. Navigate to the ML models directory:
   ```bash
   cd ML_Models
   ```
2. Create and activate a Python virtual environment:
   ```bash
   python -m venv venv
   source venv/bin/activate  # On Windows use: venv\Scripts\activate
   ```
3. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```
4. Download and place the pre-trained `DistilBERT` models in `/ML_Models/english_model` and `/ML_Models/arabic_model` directories.
5. Start the FastAPI server on port 8000:
   ```bash
   uvicorn main:app --port 8000
   ```

### 2. Set Up and Run the Web API
1. Navigate back to the root directory and locate the `API` project.
2. Update the `DefaultConnection` string in `API/appsettings.json` with your SQL Server connection details.
3. Configure your SMTP mail settings in `appsettings.json` for OTP delivery.
4. Run EF database migrations to generate the schema:
   ```bash
   dotnet ef database update --project Infrastructure/ --startup-project API/
   ```
5. Start the Web API server:
   ```bash
   dotnet run --project API/API.csproj
   ```

## API Endpoints Reference

### Authentication Endpoints
* `POST /api/auth/register` - Registers a new user account (sends OTP).
* `POST /api/auth/verify-otp` - Verifies the 6-digit OTP code to complete registration or password resets.
* `POST /api/auth/login` - Authenticates credentials and returns JWT Access Token and Refresh Token.
* `POST /api/auth/refresh-token` - Generates a new JWT Access Token using a valid Refresh Token.
* `POST /api/auth/guest` - Generates a temporary guest token for instant evaluation.
* `POST /api/auth/forgot-password` - Initiates the password recovery flow (sends reset OTP).
* `POST /api/auth/reset-password` - Updates the user password after successful OTP verification.

### Detection Endpoints
* `POST /api/detection/detect` - Main entry point to detect AI content. Accepts text or file uploads.
* `GET /api/detection/history/{userId}` - Fetches historical scans performed by the specified user.

## Testing

A comprehensive Postman collection `AI_Detector_Full.postman_collection.json` is provided in the root directory. To run tests:
1. Import the collection file into Postman.
2. Ensure your local backend servers (both .NET and Python) are running.
3. Run the requests inside the collection sequentially to test full authentication, guest access, and detection scenarios.
